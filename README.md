# TraceLens

**Live demo:** https://tracelens-7in7.onrender.com/ui

**Codex Session ID:** 019f703d-897d-73a3-8aa1-c1869ec31058

**AI Agent Debugger** — Debug AI agent workflows like you debug code.

TraceLens is an observability and debugging tool for AI agent runs. It captures every step of a multi-step agent execution, automatically detects common failure patterns (retry loops, tool-call errors, silent timeouts), and uses an LLM to generate plain-English explanations of what went wrong and what to do next — all viewable in a browser-based replay timeline.

## The Problem

AI agents fail in ways that are hard to see. A retry loop silently burns time and tokens. A tool call returns an error buried three steps deep. A step times out with no clear signal. Developers building agentic systems need a fast way to see *what happened* in a run and *why it failed*, without manually digging through logs.

## Features

- **Trace ingestion API** — a simple event schema (`agent_id`, `run_id`, `step_id`, `step_type`, `input`, `output`, `status`, `latency_ms`, `parent_step_id`) that any agent can post events to.
- **Automatic failure detection** — three built-in detectors:
  - **Retry loops** — the same tool call repeating with identical input
  - **Tool-call errors** — steps that complete with an error status
  - **Silent timeouts** — steps that exceed a configurable latency threshold with no clear resolution
- **AI-generated explanations** — each detected issue is paired with a plain-English explanation and a suggested next debugging step, generated by an LLM (Groq/Llama 3.3) and grounded strictly in the actual event data. Falls back to a rule-based explainer automatically if no API key is configured, so the app always works.
- **Replay UI** — a dark-themed, step-by-step timeline showing every event in a run, with flagged steps clearly marked in red and expandable to show the AI explanation inline.
- **Three demo scenarios included** — a research-agent retry loop, a research-agent tool error, and a mixed-failure customer-support triage run (different domain, different step types, two failure types combined) to demonstrate the system generalizes beyond a single hardcoded example.

## Getting Started

### Requirements
- Python 3.10+
- A [Groq API key](https://console.groq.com) (optional — the app runs fully offline without one, using a rule-based fallback explainer)

### Setup

```bash
git clone https://github.com/Kalpesh1Sharma/TraceLens.git
cd TraceLens
pip install -r requirements.txt
```

Create a `.env` file in the project root:
```
GROQ_API_KEY=your_key_here
```
(Optional — omit this and the app will automatically use the offline rule-based explainer instead.)

### Run

```bash
python -m uvicorn main:app --reload
```

Then open:
- **Replay UI:** [http://127.0.0.1:8000/ui](http://127.0.0.1:8000/ui)
- **API docs:** [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs)

### Try the demo scenarios

On first startup with an empty database, TraceLens automatically seeds all
three demo runs. Use the UI quick-load buttons immediately, or run the script
below when posting those same scenarios to another running API:

```bash
python toy_agent.py
```

Then in `/ui`, click any of the three "Load demo-..." quick-load buttons:
- `demo-retry-loop` — a research agent stuck retrying a search
- `demo-tool-error` — a research agent hitting a parser error
- `demo-mixed-failures` — a customer-support triage agent with a timeout and a tool error in the same run

## API Overview

| Method | Endpoint | Description |
|---|---|---|
| POST | `/traces/ingest` | Ingest one trace event |
| GET | `/traces/{run_id}` | Get a run's chronological events and reconstructed trace tree |
| GET | `/traces/{run_id}/issues` | Detect failures in a run |
| GET | `/traces/{run_id}/issues/explained` | Detect failures and attach AI-generated explanations |
| GET | `/ui` | Browser-based trace replay interface |

## Tech Stack

- **Backend:** FastAPI, Pydantic, SQLAlchemy (SQLite)
- **LLM explanations:** Groq API (Llama 3.3 70B), with an offline rule-based fallback
- **Frontend:** Vanilla HTML/CSS/JavaScript (no build step, no framework — served directly by FastAPI)
- **Testing:** Python `unittest`

## How Codex Built This

TraceLens was built end-to-end in the OpenAI Codex desktop app using GPT-5.6 (Terra), across several focused sessions using Local and Worktree-based tasks tied directly to this repository.

### Where Codex accelerated the work

Every core module — the FastAPI ingestion/retrieval API, the Pydantic trace schema, the SQLite storage layer, the three failure detectors, the Groq-backed explanation layer with its offline fallback, the full replay UI, and the third generalization-test scenario — was generated by Codex from scoped, spec-driven prompts. Each prompt described the exact schema, endpoints, and behavior needed; Codex then wrote, and in most cases self-tested, the implementation using FastAPI's test client or Python's `unittest` before handing it back. Building this manually — especially the trace-tree reconstruction logic and the multi-scenario detector test suite — would have taken meaningfully longer than the active Codex session time it actually took.

### Where key decisions were made

- **Scope**: I deliberately scoped this to a self-contained observability tool rather than a broader agent platform, so it could be fully built, tested, and demoed within the hackathon window rather than left as a partial system.
- **Provider-agnostic explanations**: I asked Codex to design the LLM explanation layer behind a swappable `Explainer` interface rather than hard-coding a single provider. This was a deliberate engineering decision to keep the system resilient — the app runs fully offline via a rule-based fallback explainer if no API key is present, and swapping providers is a one-line change.
- **Generalization over polish**: instead of stopping at one working demo, I directed Codex to build a third, structurally different scenario (a customer-support triage agent, distinct from the original research-agent examples) specifically to prove the detectors and UI generalize rather than being overfit to the first two examples.

### Debugging with Codex

Not everything worked on the first pass. After wiring up Groq, calls kept failing with an authentication error even after generating a fresh API key. Working through it with Codex-generated code and my own terminal debugging, I traced the root cause to a subtlety in how `python-dotenv` interacts with pre-existing shell environment variables — a stale `GROQ_API_KEY` from an earlier terminal session was silently taking precedence over the `.env` file. I asked Codex to add explicit `.env` loading to `main.py`, and confirmed the fix by testing the `/issues/explained` endpoint directly. Notably, the app never crashed during this — it gracefully degraded to the rule-based explainer the whole time, which validated that the resilience design worked as intended even when the integration broke upstream.

### Summary

Codex/GPT-5.6 wrote effectively all production code in this repository. My role was product and architecture direction — defining the schema, scoping each build phase, specifying failure modes and edge cases for Codex to handle, verifying every claim Codex made through manual API testing and browser checks rather than trusting generated summaries alone, and debugging the one real integration issue that came up.
